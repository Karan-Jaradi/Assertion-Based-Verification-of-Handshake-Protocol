# Assertion-Based-Verification-of-Handshake-Protocol

## Introduction

This project implements and verifies a **synchronous handshake module** in Verilog, using SystemVerilog Assertions to check request-acknowledge protocol correctness.

## Features

- Two-state FSM (`IDLE`, `WAIT`) controlling request-acknowledge handshaking
- Counter-based fixed-latency response ‚Äî acknowledgment generated after a deterministic delay
- Single-cycle `ack` pulse per request, ensuring one acknowledgment maps to exactly one request
- Synchronous design with asynchronous `rst`, triggered on `posedge clk` or `posedge rst`
- Overlapping/invalid request detection ‚Äî flags a protocol violation if `req` is asserted while not in `IDLE`
- SystemVerilog Assertions (SVA) to automatically check bounded acknowledgment delay, single-cycle `ack`, and absence of spurious `ack`
- Transaction tracking with per-request latency measurement and end-of-simulation summary

## Module Ports

### `handshake` ‚Äî Port List

| Port Name | Direction | Width  | Description                                    |
|-----------|-----------|--------|--------------------------------------------------|
| `clk`     | Input     | 1 bit  | System clock signal                              |
| `rst`     | Input     | 1 bit  | Asynchronous reset signal                        |
| `req`     | Input     | 1 bit  | Request signal from sender module                |
| `ack`     | Output    | 1 bit  | Acknowledgment signal generated by the DUT        |
| `state`   | Internal  | 2 bit  | Stores current FSM state (`IDLE` / `WAIT`)        |
| `count`   | Internal  | 3 bit  | Counter introducing delay before acknowledgment   |

## Verification

The design is verified using a self-checking Verilog testbench (`tb_handshake.v`) built around stimulus generation, real-time monitoring, transaction tracking, and SystemVerilog assertions.

**Features Verified**
- `ack` is asserted only while the DUT is in the `WAIT` state
- `ack` arrives within a bounded number of clock cycles (`1` to `5`) after a valid request
- `ack` remains high for exactly one clock cycle
- No spurious `ack` is generated without a preceding valid request
- Correct FSM transitions between `IDLE` and `WAIT`
- Correct handling of overlapping/invalid back-to-back requests

**Testbench Components**
- **Clock Generator** ‚Äî `always #5 clk = ~clk` (100 MHz, 10 ns period)
- **Reset Logic** ‚Äî asserts `rst` for a few cycles before releasing it to initialize the DUT
- **Stimulus Task** ‚Äî `send_req` task drives valid requests, plus a directed invalid overlapping-request scenario
- **Transaction Tracker** ‚Äî records the cycle of each request and its corresponding `ack`, computing per-transaction latency
- **Cycle Counter** ‚Äî tracks simulation cycles for timing and latency measurement
- **Monitor** ‚Äî prints real-time `state`, `req`, `ack`, and `count` values every clock cycle
- **SVA Assertions**:
  - `ack_wi_5_cycles` ‚Äî a valid request must be followed by `ack` within 1‚Äì5 cycles
  - `ack_one_cycle` ‚Äî `ack` must not remain high for more than one cycle
  - `no_spurious_ack` ‚Äî every `ack` must be preceded by a valid request within the last 5 cycles
- **Self-Check Summary** ‚Äî prints total requests sent, total `ack`s received, and any outstanding (unacknowledged) transactions

**Simulation Result**
- All transactions completed with a measured latency of 4 clock cycles per request
- One directed overlapping/invalid request was correctly flagged as a protocol violation
